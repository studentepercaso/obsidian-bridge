# Obsidian Bridge 0.5.4 — Companion review hardening / Hardening del companion per la revisione

## English

Obsidian Bridge 0.5.4 is a coordinated hardening release for the bundled and standalone Bridge Control companion. The companion no longer imports `node:child_process` and never starts an executable. Its optional CLI action performs only read-only metadata checks against an allowlist of known paths and reports a candidate as neither executed nor certified. The external bridge remains responsible for the definitive CLI readiness check when an allowlisted CLI-backed operation is requested. Handler registration is displayed separately inside Obsidian.

The folder picker now excludes the vault's actual `Vault.configDir` rather than assuming `.obsidian`. Shared settings migrate to version 5 so that path is also an authoritative external-bridge deny rule, including whole-vault modes. The dedicated deny is always case-insensitive, closing case-aliases when a Linux vault resides on NTFS, exFAT, or another case-insensitive volume without changing normal Linux allowlist matching. Version-2 through version-4 entries remain deny-all until their own vault records the real path, and saved folder grants intersecting it are removed. The migration can only narrow access and never invents a permission. Equivalent character-code validation replaces the control-character regular expressions flagged by automated review. The release also resolves the reported assertion, Promise listener, unsafe assignment, non-`Error` throw, unused declaration/import, timer, and CSS `:has()` findings. Standalone source and bundle checks prohibit `child_process`, `exec`, `execFile`, `spawn`, `fork`, and `shell: true` from reappearing.

Bridge Control still needs Node filesystem access to coordinate with the separate local bridge process. That access is limited to documented external stores: shared settings and their ownership lock or revocation quarantine; the read-only, size-bounded Obsidian registry and optional CLI-candidate metadata; authenticated one-time management requests; recovery backups; and metadata-only audit records. It is never used for note paths. Managed note reads and mutations remain on public Obsidian APIs: `Vault.process`, `Vault.rename`, and `FileManager.trashFile`. The external bridge's bounded exact UTF-8 note snapshots remain read-only and its create/append mutations remain on the allowlisted official CLI.

The standalone companion now has a GitHub Actions release workflow that performs locked checks, requires two byte-identical production builds, creates SLSA build-provenance attestations for `main.js`, `manifest.json`, and `styles.css`, and uploads those exact files to a retry-safe draft release. The manual backfill path refuses to attest an existing release unless the complete published asset set matches a clean rebuild of the exact tag; it creates a distinct signed historical-verification predicate instead of claiming the old assets were built by the later run. A local dry run rebuilt 0.5.3 twice identically and matched all three published files byte for byte; this does **not** claim that GitHub attestations for 0.5.3 have already been created.

Version 0.5.4 adds no permission, command, protocol field, network request, direct filesystem note write, permanent deletion, or other mutation surface. Existing permission choices are preserved except for scopes that would expose the actual configuration directory; legacy vaults must open once to complete their fail-closed version-5 migration. Install matching 0.5.4 components, reload Obsidian, refresh or reinstall the Codex plugin, and start a new Codex task.

## Italiano

Obsidian Bridge 0.5.4 è una release coordinata di hardening per il companion Bridge Control incluso e autonomo. Il companion non importa più `node:child_process` e non avvia mai eseguibili. L'azione CLI facoltativa esegue soltanto controlli di metadati in sola lettura su una allowlist di percorsi noti e indica il candidato come non eseguito e non certificato. Il bridge esterno resta responsabile della verifica definitiva quando viene richiesta un'operazione allowlistata basata sulla CLI. Lo stato di registrazione dell'handler viene mostrato separatamente dentro Obsidian.

Il selettore delle cartelle esclude ora il vero `Vault.configDir` del vault invece di presumere `.obsidian`. Le impostazioni condivise migrano alla versione 5 affinché quel percorso sia anche una regola di esclusione autorevole nel bridge esterno, comprese le modalità per l'intero vault. Il divieto dedicato ignora sempre maiuscole e minuscole: chiude gli alias di maiuscole quando un vault Linux si trova su NTFS, exFAT o un altro volume case-insensitive, senza cambiare le normali allowlist Linux. Le voci dalla versione 2 alla versione 4 restano senza accesso finché il proprio vault non registra il percorso reale; ogni ambito salvato che lo interseca viene rimosso. La migrazione può soltanto restringere l'accesso e non inventa permessi. Una validazione equivalente basata sui codici dei caratteri sostituisce le espressioni regolari sui caratteri di controllo segnalate dalla revisione automatica. La release risolve inoltre le segnalazioni relative ad assertion, listener con Promise, assegnazione non sicura, lancio di valori diversi da `Error`, dichiarazione/import inutilizzati, timer e CSS `:has()`. I controlli standalone su sorgente e bundle impediscono la reintroduzione di `child_process`, `exec`, `execFile`, `spawn`, `fork` e `shell: true`.

Bridge Control necessita ancora delle API filesystem Node per coordinarsi con il processo bridge locale separato. L'accesso è limitato agli archivi esterni documentati: impostazioni condivise e relativo lock di proprietà o quarantena per revoca; registro Obsidian di sola lettura e con limite dimensionale e metadati facoltativi del candidato CLI; richieste di gestione autenticate e monouso; backup di recupero; record audit composti solo da metadati. Non viene mai usato per i percorsi delle note. Letture e modifiche gestite delle note restano sulle API pubbliche di Obsidian: `Vault.process`, `Vault.rename` e `FileManager.trashFile`. Gli snapshot UTF-8 esatti e limitati del bridge esterno restano di sola lettura e le mutazioni create/append restano sulla CLI ufficiale allowlistata.

Il companion standalone dispone ora di un workflow GitHub Actions che esegue i controlli dal lockfile, richiede due build di produzione identiche byte per byte, crea attestazioni SLSA di provenienza per `main.js`, `manifest.json` e `styles.css` e carica quegli stessi file in una bozza di release ripetibile. Il percorso manuale di backfill rifiuta l'attestazione se l'intero insieme degli asset pubblicati non coincide con una build pulita del tag esatto; crea un distinto predicato firmato di verifica storica senza dichiarare che i vecchi asset siano stati prodotti dall'esecuzione successiva. Una prova locale ha ricostruito due volte la 0.5.3 in modo identico e ha confrontato con successo, byte per byte, tutti e tre i file pubblicati; questa frase **non** dichiara che le attestazioni GitHub della 0.5.3 siano già state create.

La versione 0.5.4 non aggiunge permessi, comandi, campi di protocollo, richieste di rete, scrittura diretta delle note via filesystem, cancellazione permanente o altre superfici di modifica. Le scelte esistenti vengono conservate salvo gli ambiti che esporrebbero la cartella di configurazione reale; ogni vecchio vault deve essere aperto una volta per completare la migrazione prudente alla versione 5. Installa i componenti 0.5.4 corrispondenti, ricarica Obsidian, aggiorna o reinstalla il plugin Codex e avvia una nuova attività Codex.
